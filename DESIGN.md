## 1. ビジョンとポジショニング

### 動機

海外 EC で住所入力(州・県の概念、表記揺れ、多言語)が面倒。日本郵便のデジタルアドレス(7桁英数字 → 住所 DB 引き)のグローバル版が欲しい。

### 戦略: 2段ロケット

- **Phase 1(現在設計中)**: 個人向け住所ウォレットツール。種まき期。認知拡大が第一目的
- **Phase 2(将来)**: 法人向け逆引き API(事前承認カプセル型、非リアルタイム用途に限定)
- Phase 1 は手段、Phase 2 が本来の目的

### ビジョンと法的立場の整合

GDA のビジョン「全世界共通の住所コード」と、法的立場「日本居住者向け、EU 拒否」の矛盾は、**メール型連合モデル**の採用によって構造的に解消されました。

- **OAP プロトコル**は全世界共通のオープン仕様
- **Postkey インスタンス**は Taketo が運営する日本居住者向けサービス
- 将来 EU 居住者向けに別の運営者が独自インスタンスを立てることが可能
- これにより「プロトコルは全世界、インスタンスは地域別」という形で矛盾が解消

### 差別化ポジション

既存の住所ウォレット(Bitwarden Identity、1Password Identities、Proton Pass Identity)との差別化:

- **多スクリプト対応**: 漢字 + ふりがな + ローマ字を一級サポート
- **国別フォーマット対応**: libaddressinput を活用した 240 カ国対応
- **専用設計**: パスワードマネージャの片手間機能ではなく、住所のために設計
- **オープンプロトコル**: 単一サービスではなく、誰でも実装・運営可能な標準
- **ゼロ知識**: 運営者(Taketo)も住所を見られない

ポジショニング: **「全世界の住所を、あなたの言語とスクリプトで」**

## 2. アーキテクチャ

### メール型連合モデル(確定)

OAP は SMTP / DNS と同じ思想に基づくメール型連合モデルを採用します。

**特性**:
- 各インスタンスは完全独立
- DNS で発見(SMTP と同様)
- インスタンス間の常時通信は不要(連合プロトコルの複雑さを回避)
- ActivityPub のような重い連合プロトコルは採用しない

**ルーティング**:
- GDA コード `taketo.yamaguchi@postkey.jp` の `@postkey.jp` 部分から DNS 経由でインスタンスを発見
- 例: `_oap._tcp.postkey.jp` の SRV レコード、または `/.well-known/oap/api` の固定エンドポイント
- 中央レジストリ不要

**3階層構造**:
1. プロトコル: OAP
2. リファレンス実装: Lodester (AGPL v3 OSS)
3. 公式インスタンス: Postkey (postkey.jp、Taketo 運営)

### 自己ホスト可能性

Lodester は自己ホスト可能なソフトウェアとして設計されます。誰でも自分の VPS や NAS で動かすことができ、独立したインスタンスを運営できます。

## 3. データモデル

vCard 4.0 + libaddressinput 準拠、独自 JSON で内部表現。

```
Person {
  schema_version,
  person_id (UUIDv4),
  names[]: {
    id,
    script (BCP 47 + ISO 15924),
    family,
    given,
    sort_as,
    pref
  },
  phones[]: {
    id,
    e164,
    type,
    extension,
    pref
  },
  emails[]: {
    id,
    address,
    type,
    pref
  },
  addresses[]: {
    id,
    type (home/work),
    custom_label,
    country,
    pref,
    fields: {
      "ja-Jpan": { O, A, C, D, S, Z, X },  // libaddressinput 8レター
      "ja-Latn": { ... }
    }
  }
}

Projection {
  projection_id,
  person_id,
  gda_code,
  label,
  includes,
  valid_until,
  revoked_at
}
```

### 主要決定

- スコープ単位: **個人**(家族・組織は対象外)
- エクスポート形式: **独自 JSON + vCard**
- 名前のスクリプト識別: **BCP 47 + ISO 15924 拡張**(例: `ja-Jpan`、`ja-Latn`)
- libaddressinput の 8 レター(N/O/A/C/D/S/Z/X)を採用
- 番地レベルのローマ字化はユーザー入力必須(libaddressinput が提供しないため)

## 4. 暗号アーキテクチャ

Bitwarden 流ゼロ知識、シンプル設計。

### アルゴリズム選定

| 用途 | アルゴリズム | 備考 |
|---|---|---|
| KDF (Key Derivation) | **Argon2id** | 600,000 iter 相当 |
| 対称暗号 | **AES-GCM-256** | ISO/IEC 18033 準拠 |
| 鍵伸長 | **HKDF-SHA256** | |
| 非対称(Phase 2) | **RSA 3072bit** | 事前承認カプセル用 |
| Secret Key | 採用しない | 1Password の二要素方式は不採用 |

### 復旧機構

**なし**(住所は記憶から再入力可能という洞察)。マスターパスワードを忘れたユーザーはデータを失うが、住所自体は記憶に残っているので新規アカウントで再入力できる。

### 暗号文ヘッダ(クリプトアジリティ)

すべての暗号文ブロブには以下のヘッダが必須:

```
{
  schema_version,
  kdf,
  cipher,
  wrapped_key_id,
  created_at
}
```

将来のアルゴリズム変更(例: ポスト量子暗号への移行)に対応可能。

### サイズ制約

- 暗号文ブロブ上限: **64 KB / ユーザー**

### バックアップ

- 頻度: 日次
- 拠点: 東日本 + 西日本の 2 箇所
- 保持期間: 30 日
- 形式: 暗号文のまま(運営者も復号不可)

### email の取り扱い

2 層構造:
- **認証用**: ハッシュ化(Argon2 等)で保存
- **通知用**: オプトインで平文保存(セキュリティ通知・アカウント削除予告のため)

## 5. GDA コード形式

### 形式(改訂版)

メール型連合モデル採用により、コードに**インスタンス識別子を含む**形式が確定しました。

**ハイブリッド形式**:
- **デフォルト**: ランダム 12 文字 Crockford Base32 + Luhn mod 32 チェックサム 2 文字 + インスタンス識別子
  - 例: `A7K29MQ3XC4F@postkey.jp`
- **カスタムハンドル**: ユーザー自選 + インスタンス識別子
  - 例: `taketo.yamaguchi@postkey.jp`

### ハンドルのルール

- ASCII 限定(英数字 + ドット + ハイフン)
- カスタムハンドルは**ドットを最低1つ含む必須**(`taketo.home`、`t.yamaguchi` 等)
- ランダム形式は `XXXX-XXXX-XXXX` のハイフン区切り固定で視覚的に区別
- 一度取得したハンドルは**解放しない**(プライバシー事故防止)
- インスタンス識別子(`@postkey.jp` 等)は必ず含む
- 公式インスタンスのユーザーは表示上 `@postkey.jp` 省略可、内部的には常に完全形

### Unicode 対応

- Unicode ハンドルは Phase 1 では非対応(homoglyph 攻撃対策)
- Phase 3 以降の検討事項として記録

## 6. 法的方針

### APPI(日本)

ゼロ知識設計が APPI「高度な暗号化」要件を満たします:
- AES-GCM-256 は ISO/IEC 18033 準拠
- 復号鍵はユーザー端末のみで生成・保持(完全分離)
- 漏えい時の報告義務・本人通知義務は免除される見込み
- 小規模事業者特例も適用
- 事前相談は自己判断、必要時のみ実施

### GDPR(EU)

Article 3(2) の「ターゲティング基準」に基づき、EU をターゲットしない運用で適用外:
- 日本語と英語のみ(EU 言語提供しない)
- EU 通貨(EUR)建ての価格表示しない
- EU TLD(.eu / .de / .fr 等)を取得しない
- EU 圏向けマーケティングを行わない
- 登録時に「EU/EEA 域外居住」チェックボックス
- household exemption(個人・家族用途)を利用規約で明記

`postkey.jp` は日本限定運営、`gda.eu` を立てたい人は別運営者がやればよい(連合モデルの利点)。

### 第三者の住所登録

家族・友人など本人以外の住所登録は GDPR の household exemption の対象として整理(個人・家族内利用に限定する旨を規約に明記)。

## 8. アナリティクス方針

1Password 流ハイブリッドモデルを参考にしつつ、より厳格に。

### Tier 1: 運用必須情報(同意不要)

- email ハッシュ(認証用)
- Protected Vault Key
- ciphertext_blob
- GDA code
- public key(Phase 2 用)
- アクセスログ(IP、7 日保持のみ、乱用防止専用)

### Tier 2: Telemetry データ(オプトイン)

- **MVP では実装しない**
- Phase 1b 以降、必要性が見えたら 1Password 方式で実装
- 実装する場合は opt-in のみ、de-identify、21 日以内削除、透明性レポート公開

### MVP のアナリティクス

サーバー側の集計カウンタのみ(個人と紐付かない):

- 総ユーザー数
- 総 GDA コード発番数
- 総ハンドル発行数
- 日次アクティブユーザー数(DAU、集計のみ)

### 測定しないもの

- 個人のセッション長・クリックストリーム
- ユーザー属性(年齢・性別・地域)
- 退会理由追跡
- A/B テスト

## 9. 非アクティブアカウント方針

## 設計判断の総括

このプロジェクトは以下の核心原則に基づいています:

1. **ユーザー主権**: ユーザーがすべてのデータの暗号鍵を握る
2. **オープンスタンダード**: プロトコルは公開、誰でも実装可能
3. **フェデレーション**: 中央集権を避け、メール型連合で分散
4. **i18n ファースト**: 多スクリプト・多国対応を一級機能として
5. **プライバシー優先**: ゼロ知識、最小限のメタデータ、透明性
6. **段階的展開**: Phase 1 で種まき、Phase 2 で本来の価値を実現
