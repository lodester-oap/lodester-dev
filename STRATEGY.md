# 認知拡大戦略の骨格

このドキュメントは Phase 1 における認知拡大戦略の骨格を定義します。各要素は実装フェーズ以降で詳細化されます。

## 戦略の構成要素(7要素)

1. ゴール設定
2. ターゲットオーディエンス
3. チャネル
4. コンテンツ戦略
5. ローンチプラン(90日)
6. 成功指標
7. アンチパターン

---

## 1. ゴール設定

**Phase 1 のゴール** — Phase 2 を起動可能にするだけの下地を作る。

### 3 年以内の目標

- **Postkey 登録ユーザー 10,000 人**(Phase 2 の B2B API を事業者に売り込める最低ライン)
- **他者運営のインスタンスが少なくとも 1 つ存在する**(プロトコルが広まっている証明)
- **OAP 仕様書がコミュニティで参照される状態**(GitHub stars 1,000+、技術ブログ言及など)

### 原則

Phase 1 は**量より質**。初期 100 人の熱狂的ユーザーが後の 10,000 人を連れてくる。

---

## 2. ターゲットオーディエンス

4 つのセグメントに優先順位をつけて段階的にリーチします。

### Tier 1: 初期コア(最初の 100 人)

- プライバシー意識の高い技術者
- OSS 文化の理解者
- Privacy Guides、r/privacy、Hacker News、日本のプライバシーコミュニティ
- Lodester の技術設計(ゼロ知識・AGPL)を評価する層

### Tier 2: 拡散層(100〜1,000 人)

- 越境 EC を頻繁に使う日本在住者
- 多言語環境の人(駐在員、国際結婚、バイリンガル)
- エンジニア・クリエイティブ業界
- テックブロガー、Qiita / Zenn 投稿者

### Tier 3: 一般層(1,000〜10,000 人)

- 海外に親戚・友人がいる人
- 国際通販ライトユーザー
- ライフハック・便利ツール好き
- この層には Tier 1/2 の推薦経由で届く

### Tier 4: アンバサダー(規模に関係なく重要)

- 技術 YouTuber / Podcaster
- プライバシー擁護者
- オープンプロトコル推進派(Matrix、Fediverse、Bluesky 界隈)
- 翻訳者・通訳者(多言語住所の痛みを直接知っている)

## 3. チャネル

### ハブ

**GitHub**(`github.com/lodester-oap/lodester` 仮)
- すべての技術情報の中心
- Issues、Discussions、Wiki を活用

### 技術発信

- **Lodester 公式ブログ**(`blog.postkey.jp` または `lodester.dev/blog`)— 月 1 本ペース
- **Qiita / Zenn** — 日本語技術記事
- **Dev.to / Medium / Hacker Noon** — 英語技術記事
- **Hacker News** — 大きなアップデート時の submission

### ソーシャル

- **Mastodon**(公式アカウント)— Fediverse との親和性を示す
- **Bluesky** — 新興プロトコルコミュニティ
- **Twitter/X** — 広いリーチ(ただし依存しない)

### コミュニティ

- **GitHub Discussions** — 主たる議論場所
- **Matrix room**(将来)— リアルタイム技術チャット
- **Discord は使わない**(プロプライエタリなので思想が合わない)

### メディア接触(ローンチ時)

- 日本: Gizmodo Japan、GIGAZINE、ITmedia、Publickey
- 国際: TechCrunch、The Register、The Verge(目標レベル)

## 4. コンテンツ戦略

月に 1 本、以下のローテーションで発信します。

### コンテンツタイプ

1. **起源の物語(Origin Story)**: なぜこれを作ったか、Taketo の動機
2. **技術ディープダイブ**: ゼロ知識設計、メール型連合、Argon2id 選定理由 等
3. **設計判断の裏話**: なぜ Lodester という名前か、なぜ Japan-only か
4. **比較記事**: 「Bitwarden Identity vs Lodester」「日本郵便デジタルアドレスとの違い」
5. **チュートリアル**: 自己ホストの方法、GDA コードの共有方法
6. **透明性レポート**: セキュリティ監査結果、令状対応統計、インフラ構成

### ローンチ前に書いておくべき基礎記事 3 本

- "Why I built Lodester"(起源の物語)
- "The OAP Protocol Explained"(技術紹介)
- "Lodester's Zero-Knowledge Architecture"(セキュリティ詳細)

これらは OSS 公開と同時に blog に並べる。

## 5. ローンチプラン(90 日計画)

### Phase 0: ステルス(OSS 公開前)

- 設計書凍結
- コアコード実装開始
- 内部テスト(Taketo 自身 + 信頼できる 1〜2 人)

### Phase 1: ソフトローンチ(Day 0〜Day 7)

- GitHub repo 公開(Day 0)
- 個人ブログで「始めました」記事
- Mastodon・Bluesky で静かに告知
- 最初の 10 人の早期採用者獲得
- 反応を見て調整

### Phase 2: オープンベータ(Day 7〜Day 30)

- 公式 Postkey インスタンス稼働
- Qiita・Zenn で技術記事投稿
- Privacy Guides コミュニティに初めて言及してもらう試み
- GitHub Issues 対応
- 最初の 50〜100 人のユーザー獲得

### Phase 3: パブリックアナウンス(Day 30〜Day 60)

- Hacker News に submission(慎重にタイミング選ぶ)
- 英語技術ブログに大型記事
- 日本メディアへのプレスリリース配信
- 初回のセキュリティホワイトペーパー公開
- 目標: 500〜1,000 ユーザー

### Phase 4: コミュニティ育成(Day 60〜Day 90)

- GitHub Discussions を活発化
- 最初のコントリビューター歓迎
- フィードバックベースの機能追加
- 3 ヶ月振り返りブログ

## 6. 成功指標

プライバシー保全型で測定可能な指標のみ。

### 主要指標

- Postkey 登録ユーザー総数(集計のみ)
- 発番された GDA コード総数
- DAU / MAU(個人を特定せず集計のみ)
- 自己ホスト検出数(DNS ベース、Taketo が把握できる範囲)

### コミュニティ指標

- GitHub stars
- GitHub forks
- オープンな Issue / PR 数
- アクティブなコントリビューター数

### メディア指標

- ブログ記事数
- 外部の技術ブログ・メディアでの言及数
- Hacker News / Reddit での投稿熱量
- Twitter/Bluesky/Mastodon のメンション数

### 測定しない指標

- 個人のセッション長・クリックストリーム
- ユーザー属性(年齢・性別・地域)
- 退会理由追跡
- A/B テスト

---

## 7. アンチパターン(やってはいけないこと)

1. **グローバル対応を過剰に誇示しない** — 「全世界共通」は OAP プロトコルの話、Postkey は日本限定
2. **パスワードマネージャとの戦争を避ける** — Bitwarden / 1Password は競合ではなく補完関係
3. **プライバシーポリシーの後付け変更をしない** — 一度公開したら基本的に変えない(ユーザーの信頼の基盤)
4. **認知拡大のためにアナリティクスを緩めない** — ゼロ知識は譲らない
5. **「早く使って!」の圧迫マーケティングをしない** — ユーザーが自分から選ぶまで待つ
6. **有料プラン・広告を Phase 1 で検討しない** — 無料・広告なし・寄付モデルのみ
7. **「Twitter で流行らせる」に依存しない** — 技術の質と口コミを核に
