# 第 2 回レビューパネル

## 開催背景

調査 1〜3(libaddressinput、ゼロ知識、APPI)が完了し、暗号アーキテクチャ・法的方針が固まったタイミングで実施。

## 主な指摘事項

### 白石(セキュリティ)

- Secret Key を不採用とする決定により、マスターパスワードのみが防衛線になる
- 弱いパスワードを使うユーザーに対する保護が薄い
- 暗号文ヘッダにメタデータ(KDF, cipher)が必須(クリプトアジリティ)
- 漏えい時の対応が不明
- email ハッシュ化により、ユーザーへの通知パスがなくなる(notification paradox)

### Priya(i18n)

- CLDR Person Names の決定を保留中
- 削除前通知の文化的要件(EU 等は事前通知が必須)
- 多文化対応の設計が浅い
- 言語環境ごとのフォールバック戦略

### Marcus(データ設計)

- マスターパスワード変更時の再暗号化フローが未設計
- GDA コードがマスターパスワード変更で変わるか不明
- 暗号文ブロブのサイズ上限が未設定
- Phase 2 の RSA 鍵管理(誰が生成、いつローテート)
- スキーマバージョニング戦略

### 田村(PM)

- Phase 2 のリアルタイム要求は EC チェックアウトには不可能
- Phase 1 が成立するためにはブラウザ拡張または自動入力が必要
- 既存パスワードマネージャ(Bitwarden、1Password)との競合
- フォーム検出はパスワードマネージャでも難題
- マーケティング予算も既存ユーザーベースもない

### 黒木(懐疑的)

- スコープが**膨張している**(縮小していない)
- ゼロ知識 + 個別 RSA + AGPL + 自己ホスト + OSS ガバナンス + 法的対応 + 文書整備
- これは 3〜5 人のフルタイムチームの仕事
- 個人開発で 1 人で進めるのは非現実的
- まだ 1 行もコードが書かれていない
- AGPL の効力は個人プロジェクトでは象徴的(訴訟リソースなし)
- 大量の文書負担(安全管理措置、インシデント対応、ホワイトペーパー、自己ホストガイド、透明性レポート)
- 「ステートメントを書く時間があったらコードを書け」

## 主な決定

第 2 回の結果として:

### 直接的決定(Claude が判断)

1. **暗号文ヘッダにメタデータを必須化**: `{schema_version, kdf, cipher, wrapped_key_id, created_at}`
2. **Luhn mod 32 チェックサム採用**: GDA コードのチェックサム
3. **email を 2 層構造に**: 認証用ハッシュ + オプトイン平文通知用
4. **ブロブサイズ上限 64 KB / ユーザー**
5. **バックアップ: 日次、東日本+西日本 2 拠点、30 日保持、暗号文のまま**
6. **ZFS RAID-Z1 採用**(後の作業で検討予定)

### スコープに関する重要な決定

**Taketo の判断**: 黒木の「スコープを激減させろ」を退け、「**折衷案**」を採用。

> 「依存関係は入念に洗い出しつつ、詳細度は実装可能最小限」

ゲートを通すために必要な項目はすべて設計するが、各項目の詳細度は実装可能な最小レベルに留める。

### Phase 2 の再定義

田村の指摘を受けて、Phase 2 を以下のように再定義:

- **リアルタイム逆引きを諦める**: EC チェックアウトには対応しない
- **事前承認カプセル方式**: ユーザーが事前承認した相手のみが住所を引ける
- **非リアルタイム用途に限定**: 定期購読、年一回の物販、文通等
- 成功すれば Phase 2 を本格化、失敗すれば Phase 1 のみで終わってもよい

### 認知拡大の重視

田村の「ユーザー獲得経路がない」を受けて:

- **Phase 1 の本来目的を再定義**: 個人ツールではなく、Phase 2 へ向けた**認知拡大**の手段
- 認知拡大戦略を凍結ゲートに追加
- ターゲットユーザーを明示化
- ブランディング・命名の重要度上昇

### OSS 公開の前倒し

- **設計段階から OSS 公開**: コミュニティ早期形成
- AGPL v3 確定
- GitHub 公開準備

## 残る課題(第 2 回時点)

- ビジョン(全世界共通)と法的立場(日本限定)の矛盾
- プロジェクト命名
- 認知拡大戦略の具体化
- GDPR 対応方針
- 既存住所ウォレットとの差別化の実証(調査 7)

これらは第 3 回および追加調査で解決される。
