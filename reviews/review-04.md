## 各レビュアーの総括

### 白石 さやか(★★★★☆)

**良かった点**:
- ゼロ知識設計の一貫性が最後まで維持された
- Bitwarden 方式ログインは堅実な選択
- アナリティクスのガードレール(集計のみ、PII 不記録)が明文化された
- Reproducible Builds 採用は「信頼より検証」の精神と整合

**実装フェーズへの注意事項**:

1. **暗号実装コードは AI 生成に頼りすぎない**
   - AES-GCM の nonce 生成、Argon2id のパラメータ、HKDF のコンテキスト文字列などは微妙な誤りがセキュリティ崩壊を招く
   - 必ず `golang.org/x/crypto` の公式ドキュメントと照合
   - `gosec` の警告は全て潰す

2. **セッショントークンの設計が未定**
   - JWT vs opaque token
   - 推奨: **opaque token**(JWT は実装ミスが多い、取り消しが困難)
   - 実装時に決定

3. **メタデータリーク経路の監査**
   - 実装中に「このログには PII が漏れていないか?」を常にチェック

**MVP スコープへの意見**: 段階的実装に賛成。MFA なしでも OK、ただし Phase 1b 最初に入れるコミットは明記すべき

### Priya Raman(★★★★☆)

**良かった点**:
- libaddressinput の採用で 240 カ国対応の基盤ができた
- 多スクリプト名前(漢字 + ふりがな + ローマ字)のデータモデル
- メール型連合により「日本で作って世界へ」の物語が無理なく成立
- 命名(OAP/Lodester/Postkey)が中立的で国際性を損なわない

**懸念事項**:

1. **libaddressinput の日本以外のデータ品質**
   - 実装時にテストで確認すべき国: 米、英、独、仏、韓、中、台、印
   - ローマ字化の品質は libaddressinput の範囲外(ユーザー入力)

2. **非 ASCII ハンドル未対応**
   - Unicode 正規化・homoglyph 攻撃・URL エンコーディング問題がある
   - 設計通り Phase 3 以降でよい

**MVP スコープへの意見**: 段階的実装に強く賛成。MVP は日本語 UI のみで十分。「日本で生まれ、日本で愛されてから世界へ」という物語と整合。英語 UI は Phase 1b の早期に追加

### Marcus Holme(★★★★★)

**良かった点**:
- **メール型連合モデルはエレガントな解決**(分散 vs Phase 2 の逆引き API 両立問題を解いた)
- DATABASE_URL 抽象化により、公式インスタンスと自己ホストが同じコードベースで共存
- `sqlc` + PostgreSQL の選定は型安全性と実用性のバランスが良い
- OAP プロトコル仕様書アウトラインの 13 章構成は他者実装可能なレベル

**実装フェーズへの注意事項**:

1. **DB スキーマバージョニング戦略**
   - マイグレーションは前方・後方互換性を意識
   - `goose` または `golang-migrate` を実装時に決定
   - 初期スキーマは `v0001` で始める

2. **インデックス設計**
   - `email_hash` にユニークインデックス
   - `handle` にもユニークインデックス(大文字小文字区別なし)
   - `vault_updated_at` に通常インデックス(バックアップ戦略のため)

3. **OAP 仕様書本文の執筆時期**
   - 実装と並行して書く(先に書くと実装ミスマッチが発生)
   - 最初は実装の「自動ドキュメント」レベルで OK
   - 他者実装を受け入れるタイミング(Phase 1b 後半)までに完成

**MVP スコープへの意見**: 段階的実装に賛成。**データモデルは最初から全機能分を定義**しておくこと(後から追加はマイグレーション負荷)。「データモデル最初から、機能は段階的に」が推奨

### 田村 慎一(★★★★☆)

**良かった点**:
- 認知拡大戦略の骨格が形になっている
- 3 階層命名は認知拡大に有利
- フェーズ分離(Phase 1 で認知、Phase 2 で収益)の戦略が明確
- セーフティネット(チェックポイント方式)の導入は現実的

**懸念事項**:

1. **Taketo 側の宿題が未完**
   - `postkey.jp` ドメイン取得可能性
   - Postkey の商標調査
   - `lodester` OSS 空き確認
   - 勤務先の副業規程確認
   - **実装開始前に必ず片付ける**

2. **MVP → Phase 1b → Phase 2 の時期の明確化**
   - いつ Phase 1b に移るか
   - いつ Phase 2 検討を始めるか
   - 明確な「発動条件」が必要

3. **ローンチ後のフィードバックループ**
   - 早期ユーザーからどう意見を集めるか
   - GitHub Discussions が主か
   - **MVP ローンチ前に決めておく**

**MVP スコープへの意見**: 段階的実装に強く賛成。「小さく出して育てる」は OSS の王道。推奨 MVP 範囲はコア機能 + 共有機能 + vCard、目標は 3〜5 ヶ月でのローンチ

**痛烈な指摘**:

   - 過去の技術レビューから繰り返し警告してきた

## Taketo の最終決定

### MVP スコープ

**推奨案を採用**: MVP はコア機能 + 共有機能(QR コード)+ vCard エクスポート

**Phase 1b 第 1 波**: カスタムハンドル、TOTP MFA、英語 UI

**Phase 1b 第 2 波**: WebAuthn、ステータスページ、アクセス履歴、追加言語

**Phase 2 検討**: 登録 1,000 人到達時に評価

### 目標時期

MVP 公開を 3〜5 ヶ月、上限 6 ヶ月で実現する。AI 活用前提。

### 修正履歴

- 初回決定(第 3 回レビュー後): フル案(12 ヶ月目標)
- 訂正(第 4 回レビュー時): Taketo の意図が「段階的実装」であったことが明確化され、推奨案へ変更
