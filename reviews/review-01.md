# 第 1 回レビューパネル

## 開催背景

データモデルの初期設計が固まったタイミングで、5 人の擬似レビュアーによる横断的な批評を実施。

## レビュアー

| 名前(仮名) | 役割 |
|---|---|
| 白石 さやか | セキュリティ・プライバシー専門 |
| Priya Raman | 国際化・多言語対応専門 |
| Marcus Holme | データアーキテクト |
| 田村 慎一 | プロダクトマネージャ |
| 黒木 健司 | 懐疑的シニアエンジニア |

## 主な指摘事項

### 白石(セキュリティ)

- ゼロ知識アーキテクチャの主要な課題は鍵管理
- 復旧パスを持たないと、ユーザー喪失リスクが高い
- 一方で復旧パスを持つとゼロ知識性が崩れる根本的トレードオフ
- email を平文で保持するか、ハッシュ化するかで挙動が変わる

### Priya(i18n)

- スクリプト識別を BCP 47 + ISO 15924 で行うべき
- 番地レベルのローマ字化は libaddressinput では提供されないため、ユーザー入力が必要
- CLDR Person Names の扱いをどうするか
- 名前の sort_as フィールドの扱い

### Marcus(データ設計)

- GDA コードと住所の関係が不明確
- 住所が更新されたとき、コードは同じか変わるか
- 「個人」スコープにすると、ある住所だけ共有という機能が弱くなる
- script の persistence(個人レベル vs 住所レベル)

### 田村(PM)

- このサービスを実際に使う人は誰か
- 既存の住所機能(パスワードマネージャ等)との差別化は何か
- ユーザー獲得経路がない
- マーケティング戦略がない

### 黒木(懐疑的)

- スコープが大きすぎる
- 完璧な設計を追い求めて何も作れない waterfall trap
- 最小限の MVP を先に出すべき
- AGPL v3 の運用負担を理解しているか

## 主な決定

第 1 回の結果として:

1. **スコープ単位を「個人」に確定**(家族・組織は Phase 2 以降)
2. **独自 JSON + vCard エクスポート方式を採用**
3. **GDA コード形式に Base32 を採用**(後にハイブリッド方式に変更)
4. **次の調査を順序立てて実施することを決定**:
   - 調査 1: libaddressinput
   - 調査 2: ゼロ知識アーキテクチャ
   - 調査 3: APPI
   - 調査 4: GDPR
   - 調査 5: Signal Protocol(Phase 2 用、保留)
   - 調査 6: Web プッシュ(Phase 2 用、保留)
   - 調査 7: 既存住所ウォレット

## 残る課題(第 1 回時点)

- 復旧機構の有無
- email の格納方式
- マスターパスワード変更時の再暗号化フロー
- 差別化ポジションの実証
- 認知拡大戦略

これらは後続の調査・レビューで解決される。
