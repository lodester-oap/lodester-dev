# 意思決定ログ

このドキュメントは設計フェーズ中に行われた主要な決定を時系列で記録したものです。

## 形式

各エントリは以下の形式で記述されています:

```
### [DECISION-NNN] 決定タイトル
- **日付**: YYYY-MM-DD
- **コンテキスト**: なぜこの決定が必要だったか
- **選択肢**: 検討された選択肢
- **決定**: 採用された方針
- **根拠**: 決定の理由
```

## 設計初期(プロジェクト立ち上げ)

### [DECISION-001] プロジェクトの基本方針

- **コンテキスト**: 海外 EC で住所入力の手間を解消する手段が必要
- **決定**: 日本郵便デジタルアドレスのグローバル版を作る
- **根拠**: 既存サービスは多スクリプト・多国対応が不十分

### [DECISION-002] 2 段ロケット戦略

- **コンテキスト**: 個人向けと法人向けのどちらを先に作るか
- **選択肢**:
  - A: 法人向け(Phase 2)を先行
  - B: 個人向け(Phase 1)で種まきしてから法人向けへ
- **決定**: B を採用。Phase 1 = 個人向けで認知拡大、Phase 2 = 法人向け逆引き API
- **根拠**: 法人向けには既存ユーザー数が必須。個人ユーザーが先に必要

## データモデル設計

### [DECISION-003] スコープ単位を「個人」に限定

- **コンテキスト**: 家族・組織アカウントを Phase 1 で扱うかどうか
- **決定**: Phase 1 は**個人スコープのみ**。家族・組織は Phase 2 以降

### [DECISION-004] vCard 4.0 + libaddressinput を採用

- **コンテキスト**: 住所データの内部表現と外部交換形式の選定
- **決定**:

  - エクスポート: vCard 4.0 + 独自 JSON
  - 住所フィールド: libaddressinput の 8 レター(N/O/A/C/D/S/Z/X)
- **根拠**: vCard は標準として広く実装されている、libaddressinput は 240 カ国対応で Apache 2.0

### [DECISION-005] 名前のスクリプト識別に BCP 47 + ISO 15924 を使用

- **決定**: `ja-Jpan`(日本語・漢字仮名)、`ja-Latn`(日本語・ローマ字) のような形式
- **根拠**: 国際標準。CLDR / Unicode 規格と整合

## 暗号アーキテクチャ

### [DECISION-006] Bitwarden 流ゼロ知識を採用

- **選択肢**:
  - A: Bitwarden 流(マスターパスワードのみ)
  - B: 1Password 流(マスターパスワード + Secret Key)
  - C: Proton 流(PGP ベース)
- **決定**: A(Bitwarden 流、シンプル)
- **根拠**: 住所は記憶から再入力可能なので、復旧不能リスクの重さが軽減される

### [DECISION-007] アルゴリズム選定

- **決定**:
  - KDF: Argon2id(600,000 iter 相当)
  - 対称暗号: AES-GCM-256
  - 鍵伸長: HKDF-SHA256
  - 非対称(Phase 2): RSA 3072bit
- **根拠**: 業界標準、ISO/IEC 準拠

### [DECISION-008] Secret Key 不採用

- **決定**: 1Password 風の Secret Key は採用しない
- **根拠**: シンプルさ優先、住所データは記憶可能なのでリスク低

### [DECISION-009] 復旧機構なし

- **決定**: マスターパスワードを忘れたらデータ喪失。復旧パスは提供しない
- **根拠**: ゼロ知識を完全保証するため。住所は再入力可能

### [DECISION-010] 暗号文ヘッダにメタデータ必須

- **決定**: 全暗号文ブロブに `{schema_version, kdf, cipher, wrapped_key_id, created_at}` ヘッダ必須
- **根拠**: クリプトアジリティ(将来のアルゴリズム変更に対応)

### [DECISION-011] ブロブサイズ上限 64KB

- **決定**: 暗号文ブロブ上限 64 KB / ユーザー
- **根拠**: 通常用途で十分、悪用防止

### [DECISION-012] バックアップ方針

- **決定**: 日次、東日本+西日本 2 拠点、30 日保持、暗号文のまま
- **根拠**: 災害対策と実用性のバランス

### [DECISION-013] email を 2 層構造に

- **決定**:
  - 認証用: ハッシュ化
  - 通知用: オプトインで平文(セキュリティ通知・削除予告のため)
- **根拠**: ゼロ知識と運用必要性のバランス

## 法的方針

### [DECISION-014] APPI 高度な暗号化例外を活用

- **決定**: ゼロ知識設計を APPI 「高度な暗号化」要件として利用
- **根拠**: 漏えい時の通知・報告義務が免除される。AES-GCM-256 は ISO/IEC 18033 準拠
- **参照**: research/03-appi.md

### [DECISION-015] GDPR 適用回避(Japan-only ターゲティング)

- **決定**: GDPR の Article 3(2) ターゲティング基準に基づき、EU 居住者をターゲットしない運用で適用外
- **対策**:
  - 日本語と英語のみ
  - EU 通貨/TLD/マーケティングなし
  - 登録時に EU/EEA 域外居住チェックボックス
  - household exemption 活用
- **根拠**: 個人開発で GDPR 完全コンプライアンスは現実的でないため
- **参照**: research/04-gdpr.md

## アーキテクチャ転換

### [DECISION-016] メール型連合モデルを採用

- **コンテキスト**: 「全世界共通」のビジョンと「日本限定」の法的立場の矛盾
- **選択肢**:
  - A: ビジョンを縮める(日本限定で完結)
  - B: GDPR 完全対応(費用・労力的に却下)
  - C: フェーズで分離(現在は日本、将来世界)
  - D: メール型連合モデル(プロトコルは全世界、インスタンスは地域別)
  - E: 現行維持
- **決定**: D(メール型連合モデル)
- **根拠**:
  - SMTP / DNS と同じ思想で、矛盾が構造的に解消
  - ActivityPub 連合より圧倒的に軽量
  - DNS が routing を担うので独自プロトコル不要
- **参照**: reviews/review-03.md

### [DECISION-017] 自己ホスト可能性を前面化

- **決定**: Lodester は自己ホスト可能なソフトウェアとして設計、自己ホストを「特典」ではなく「中核」に
- **根拠**: メール型連合モデルの本質。プライバシー意識層への訴求

## 命名

### [DECISION-018] 3 階層命名(完全分離)

- **決定**:
  - プロトコル: **OAP** (Open Address Protocol)
  - 実装: **Lodester**(北極星、コンパス以前のナビ)
  - 公式インスタンス: **Postkey** (`postkey.jp`、郵便の鍵)
- **根拠**:
  - SMTP / Postfix / Gmail のような明確な階層
  - OAP = 技術的・中立的・記述的
  - Lodester = 物語性・詩的・OSS 名らしい
  - Postkey = 機能性・覚えやすさ・サービス名らしい

### [DECISION-019] GDA コード形式

- **決定**:
  - ハイブリッド: ランダム 12 文字 Base32 + Luhn または カスタムハンドル
  - 必ずインスタンス識別子を含む(例: `taketo.yamaguchi@postkey.jp`)
  - ハンドルは ASCII 限定、ドット必須
  - 一度取得したハンドルは解放しない
  - 公式インスタンスのユーザーは表示上 `@postkey.jp` 省略可
- **根拠**: メール型連合の routing に必要

## プロトコル仕様アウトライン凍結時の決定

### [DECISION-026] インスタンス発見方式

- **コンテキスト**: クライアントが GDA コードからインスタンスのエンドポイントを発見する方式
- **選択肢**: SRV レコード / well-known URI / TXT レコード
- **決定**: well-known URI 方式(`https://{instance}/.well-known/oap`)
- **根拠**: HTTPS だけで完結、ブラウザから読める、WebFinger / Mastodon と同じ思想、JSON で拡張可能

### [DECISION-027] インスタンス識別子の制約

- **コンテキスト**: インスタンス識別子としてどのようなホスト名を許容するか
- **決定**: 公開 DNS で解決可能なホスト名のみ(MUST)
- **根拠**: `.onion` 等の代替ネットワークは Phase 3 以降の拡張仕様で扱う。MVP 複雑度を抑える

### [DECISION-028] 認証方式(ログイン)

- **コンテキスト**: ゼロ知識アーキテクチャでのパスワード認証方式
- **選択肢**: SRP / OPAQUE / Bitwarden 方式(KDF 派生ログインハッシュ)
- **決定**: Bitwarden 方式
- **根拠**:
  - 個人開発で実装可能なシンプルさ
  - Bitwarden / Proton Pass で大規模実証済み
  - 任意の言語・フレームワークで実装可能(プロトコル普及にとって重要)
  - 真の PAKE と比較した実用上の安全性差はほぼゼロ
  - マスターパスワードもマスターキーも MUST NOT 送信

### [DECISION-029] MFA 方針

- **コンテキスト**: 多要素認証のサポートレベル
- **決定**:
  - TOTP (RFC 6238): 実装が提供 MUST
  - WebAuthn / Passkey: 実装が提供 SHOULD
  - バックアップコード: MFA 有効化時に発行 MUST(リカバリ機構なし設計のため)
  - SMS / Email OTP: MAY(非推奨)
  - ユーザーによる MFA 有効化: 任意(強制 MUST NOT)
- **根拠**: SHOULD レベルが現代的サービスの標準。リカバリなし設計のためバックアップコードは絶対必要

### [DECISION-030] API 形式

- **コンテキスト**: API のプロトコル形式選択
- **選択肢**: REST / RPC / GraphQL
- **決定**: REST 風 HTTP API、JSON 交換
- **根拠**: 標準的、誰でも理解できる、HTTP セマンティクスをそのまま使える、curl/ブラウザでデバッグ可能、OpenAPI 3.1 で機械可読仕様化

### [DECISION-031] API バージョニング

- **コンテキスト**: API バージョンの表現方法
- **選択肢**: URL パス / HTTP ヘッダ / メディアタイプ
- **決定**: URL パス方式(`/api/v1/...`)
- **根拠**: 最もシンプル、可視性高、Stripe / GitHub / Twitter / Slack が採用、ログ・監視で一目瞭然

### [DECISION-032] ブロックリスト機構の標準化レベル

- **コンテキスト**: 悪意あるインスタンスをブロックする機構を仕様で必須とするか
- **決定**: MAY(任意機能)
- **根拠**: Phase 1 では悪意あるインスタンスがほぼ存在しない。インスタンス運営者の自由を保つ。Phase 2 で乱用が見えたら SHOULD への昇格を検討

## インフラ構成凍結時の決定

### [DECISION-033] クラウド事業者

- **コンテキスト**: 公式 Postkey インスタンスをどこでホストするか
- **選択肢**: Sakura VPS / ConoHa VPS / Linode / AWS Lightsail / GCP / OCI
- **決定**: Linode 東京リージョン
- **根拠**: 開発者体験良好、Managed DB/Object Storage/LB 統合、東京リージョンで低レイテンシ。APPI 整合性はゼロ知識アーキテクチャと日本リージョン限定運用で確保。Linode は委託先として扱う

### [DECISION-034] バックエンド言語

- **コンテキスト**: Lodester のリファレンス実装に使う言語
- **選択肢**: Go / Rust / TypeScript / Python / Elixir
- **決定**: Go
- **根拠**:
  - 学習カーブ緩やか(本業傍らで習得可能)
  - シングルバイナリで運用が究極にシンプル
  - 他者が読みやすい = 参照実装として理想的
  - 標準ライブラリ充実、暗号処理ライブラリ成熟
  - 完成確率が最も高い

### [DECISION-035] データベースと接続抽象化

- **コンテキスト**: 公式インスタンスでマネージド DB を使いたい、自己ホストでは同梱型が楽
- **決定**:
  - PostgreSQL 16+ のみサポート(SQLite は Phase 2 以降検討)
  - `DATABASE_URL` 環境変数による接続抽象化
  - 公式: Linode Managed Database
  - 自己ホスト: Docker Compose または同居型 PostgreSQL
- **根拠**: 同一コードベースで公式・自己ホスト両方に対応。12-Factor App 原則に準拠

### [DECISION-036] 外部監視

- **コンテキスト**: Postkey の稼働を外部から監視する方法
- **選択肢**: BetterStack / UptimeRobot / Uptime Kuma(セルフホスト) / Linode 標準のみ
- **決定**: BetterStack Uptime(無料プラン)
- **根拠**: 10 monitors まで無料、モダン UX、ステータスページ自動生成、個人運営で手間なし

### [DECISION-037] OAP プロトコル標準監視エンドポイント

- **コンテキスト**: 監視機能を OAP プロトコルの一部として標準化するか
- **決定**: 2 つのエンドポイントを追加
  - `/.well-known/oap/status` (SHOULD): 認証不要、生存確認と機能一覧
  - `/admin/metrics` (MAY): 管理者認証必須、Prometheus 形式メトリクス
- **根拠**: 標準化により任意の外部監視ツールが OAP インスタンスを同じ方法で監視可能。プライバシー保全のため `/status` はユーザー数等の集計値を返さない

### [DECISION-038] デプロイ方式

- **コンテキスト**: MVP でのデプロイ自動化レベル
- **選択肢**: 完全手動 SSH / GitHub Actions 手動トリガー / タグ push 自動デプロイ
- **決定**: GitHub Actions 手動トリガー(`workflow_dispatch`)
- **根拠**:
  - 明示的デプロイ操作で安全性を確保
  - ロールバック手順が自動化
  - デプロイ履歴が GitHub に残る
  - 個人運営で現実的な複雑度
  - Phase 1b でタグ自動デプロイに進化させやすい

### [DECISION-039] Reproducible Builds

- **コンテキスト**: バイナリの再現性を保証するか
- **決定**: MVP から Reproducible Builds を採用
- **根拠**:
  - 「信頼ではなく検証」の精神と合致
  - 誰でもソースから同じバイナリを再現可能
  - Supply chain security の基盤
  - Tailscale / NixOS 等の先行例あり
  - Go なら `GOFLAGS="-trimpath -mod=readonly"` 等で実現容易

### [DECISION-040] MVP 実装範囲(段階的実装)

- **コンテキスト**: MVP にどこまで機能を含めるか
- **選択肢**: 最小案(コア機能のみ)/ 推奨案(コア + 共有 + vCard)/ フル案(全機能)
- **決定**: 推奨案
  - MVP = コア機能 + 共有機能(QR コード)+ vCard エクスポート
  - カスタムハンドルと MFA は Phase 1b 第 1 波へ
  - 英語 UI は Phase 1b 第 1 波へ
  - WebAuthn は Phase 1b 第 2 波へ
- **根拠**:
  - 段階的実装は OSS の王道(「小さく出して育てる」)
  - 個人運営で現実的な 3〜5 ヶ月での公開目標
  - 設計は全機能を想定して固めてあるので、Phase 1b での追加は容易
  - データモデルは最初から全機能分を定義する(Marcus 推奨)
- **セーフティネット**:
  - チェックポイント方式(M1〜M5)で各段階の進捗を評価
  - 遅延時(予定の 70% 未満)は対応を段階的に調整
  - Lite MVP 案(2〜3 ヶ月で共有/vCard なし)を最終手段として用意
- **目標時期**: 3〜5 ヶ月(AI 活用前提)、上限 6 ヶ月
- **修正履歴**:
  - 初回決定: フル案(12 ヶ月目標)
  - 修正: Taketo の意図の明確化により段階的実装の推奨案へ変更(第 4 回レビュー後)

### [DECISION-041] 実装名の変更(Lodestar → Lodester)

- **コンテキスト**: 設計フェーズ完了後、Taketo 側の宿題確認の中で、当初実装名として選定していた「Lodestar」が既存 OSS プロジェクトと深刻に競合していることが発覚
- **発覚した競合**:
  - **ChainSafe Lodestar**: Ethereum Consensus Client(GitHub stars 1,400+、forks 425+、Apache 2.0、ChainSafe Systems が運営、月次リリース、独自ドメイン `lodestar.chainsafe.io` 等)
  - 関連: `github.com/regalias/lodestar`(Go テキスト検索ライブラリ)、`Lodestar-Forge`(レッドチーム向け)、その他多数
- **競合の深刻度**: 高
  - 検索競合(開発者が「Lodestar」を検索すると ChainSafe Lodestar が圧倒的に上位)
  - ブランド混乱(技術記事・SNS で常に曖昧)
  - 認知拡大に直接的な不利
  - 商標リスク(ChainSafe が登録していれば法的問題)
- **検討した代替候補**:
  - **Astra**: 多数の競合(ASTRA Toolbox、ASTRA-sim、AstraExt、DataStax Astra 等)→ 却下
  - **Sidus**: 中規模競合複数(ドローン、AI、voxel game)→ △
  - **Estel**: 直接競合なし、ただし Estelle/Estrella/Stella と混同リスク → ◎
  - **Sidera**: ESA Gaia 関連 + Sidero(Kubernetes 系)混同リスク → △
  - **Cairn、Helm、Atlas、Beacon、Compass、Stella**: 全て大規模競合あり → 却下
- **決定**: **Lodester**(ロデスター)
- **根拠**:
  - **OSS 競合なし**(Web 検索でヒットせず、Lodder Software 等は綴り・分野とも別)
  - Lodestar の音韻と意味を最大限引き継ぐ(`-r` を `-er` に変更)
  - 「Lodestar(導きの星)を辿る者」という造語的解釈が可能

### [DECISION-049] 暗号文ヘッダスキーマ (Ciphertext Header)

- **日付**: 2026-04-08
- **コンテキスト**: クリプトアジリティ確保のため、暗号文に自己記述型ヘッダを付ける
- **決定**: ワイヤーフォーマット `[4-byte header_len BE][header JSON][ciphertext]`
- **ヘッダフィールド**:
  - `v`: スキーマバージョン (現在 1)
  - `alg`: 暗号アルゴリズム (`aes-gcm-256`)
  - `kdf`: KDF 名 (`argon2id`)
  - `kdf_params`: KDF パラメータ (memory, iterations, parallelism)
  - `nonce`: AES-GCM nonce (12 bytes, base64url)
  - `ct_len`: 暗号文長 (bytes)
- **根拠**: 将来のアルゴリズム変更時に旧データも復号可能にする

### [DECISION-051] 楽観的ロック (Optimistic Locking)

- **日付**: 2026-04-08
- **コンテキスト**: 複数デバイスからの同時ボールト更新時のデータ競合防止
- **決定**: vaults テーブルに `version INTEGER` カラム、更新時にバージョン一致を要求
- **方針**:
  - 初回作成: version = 1
  - 更新: `WHERE version = $expected` + version 自動インクリメント
  - バージョン不一致: 409 Conflict + `VERSION_CONFLICT` エラーコード
  - クライアントは再取得 → 再暗号化 → 再送信
- **根拠**: 悲観的ロックはコネクション保持が必要で個人運用には重すぎる

### [DECISION-052] Person / Address データの保管方針

- **日付**: 2026-04-09
- **コンテキスト**: M4 で Person(人物)と Address(住所)の永続化方式を決める必要がある。サーバー側 DB に平文で保管するか、Vault の暗号文に含めるか
- **選択肢**:
  - A: すべて Vault に入れる(サーバー DB は最小メタデータのみ)
  - B: 一部(氏名・住所行)を Vault、一部(国コード等)をサーバー DB に
  - C: すべてサーバー DB に暗号カラムとして保管
- **決定**: A(すべて Vault に入れる)
- **方針**:
  - `persons` テーブルは `id`, `user_id`, `created_at`, `updated_at` のみの最小構成(Marcus 設計)
  - 氏名(latin/native/phonetic)、住所 8 レター、電話番号、メモ等はすべて Vault JSON の中
  - サーバーは Person の存在と所有者しか知らない(ゼロ知識)
  - GDA コードのみ別途 `gda_codes` テーブルに登録(公開識別子として扱うため)
- **根拠**:
  - Lodester の哲学(ゼロ知識)を貫く。サーバー運営者が令状対応でも氏名・住所を渡せない状態にする
  - M3 レビューで白石が明確に要求「Person/Address は完全に Vault、サーバー側テーブルに機密情報を絶対入れない」
  - Marcus の参照整合性要件(ID 参照のみサーバー側で管理)を満たす

### [DECISION-053] GDA コード仕様

- **日付**: 2026-04-09
- **コンテキスト**: Global Distinct Address(世界的に一意な住所識別子)の符号化方式
- **決定**:
  - 符号化: Crockford Base32(I/L/O/U を除いた 32 文字集合)
  - 長さ: 11 文字 + Luhn mod 32 チェックサム 1 文字 = 合計 12 文字
  - 表示形式: `XXXX-XXXX-XXXX`(4-4-4 ハイフン区切り)
  - 生成源: `crypto/rand`(55 bit エントロピー、乱数衝突確率は無視可能)
- **理由**:
  - Crockford Base32 は人間が誤読しにくい(0/O, 1/I/L の混同回避)
  - Luhn mod 32 は 1 文字誤入力を 100% 検出、隣接 2 文字入れ替えを 96.875% 検出
  - 4-4-4 形式は電話番号・クレジットカードと同じ認知チャンクで音読可能
  - 55 bit は SHA-1 の衝突空間より遥かに小さいが、運用上 10^8 個までの発行を想定するため十分
- **制約**: GDA コードは「公開 ID」。個人情報を含まないので Vault 外に保管可能

### [DECISION-054] Address データモデル(libaddressinput + 多スクリプト)

- **日付**: 2026-04-09
- **コンテキスト**: 240 カ国対応を目指さない現実路線の住所スキーマ策定
- **決定**:
  - フィールド: libaddressinput 8 レター(N/O/A/C/D/S/Z/X)を踏襲
    - N = Name(受取人名)
    - O = Organization(会社名、optional)
    - A = Address lines(番地・建物名、配列)
    - C = City/Locality(市区町村)
    - D = Dependent locality(町名・大字、optional)
    - S = State/Region/Prefecture(都道府県)
    - Z = Zip/Postal code
    - X = Sorting code(optional、仏国 CEDEX 等)
  - 多スクリプト対応: 各 Address に `script` フィールド(`ja-Jpan`, `ja-Latn` 等の BCP 47)
  - 1 つの Person が複数 Address を持てる(日本語版 + ローマ字版など)
  - 国コード: ISO 3166-1 alpha-2(`JP`, `US`, `DE`)
- **対応国の優先順位**:
  - Phase 1 MVP: 日本 + 主要 5 カ国(US, DE, FR, UK, AU)のバリデーションを重点実装
  - 残り 234 カ国: 入力は受け付けるがバリデーションは最小限(郵便番号形式チェックのみ等)
- **根拠**:
  - libaddressinput は Google が Apache 2.0 で公開、世界で最も広く採用されている住所スキーマ
  - 240 カ国完璧対応は M4 では非現実的(Tamura の指摘:「完璧より完成」)
  - 多スクリプト対応は Priya の必須要件(日本語圏ユーザーの本国/海外配送需要)

### [DECISION-055] ケイパビリティ URL は URL フラグメント方式

- **日付**: 2026-04-09
- **コンテキスト**:
  M5 の中核機能「共有リンク経由で他者に住所を 1 枚渡す」を実装するにあたり、
  リンクに埋め込む復号鍵をどこに置くかを決める必要がある。Lodester は
  ゼロ知識プロジェクトなので、サーバーが鍵を見られる選択肢は原則 NG。
- **選択肢**:
  1. **URL フラグメント方式**(`https://.../share/{id}#k=<key>`)
     - Bitwarden Send / Firefox Send が採用している方式
     - `#` 以降はブラウザが決してサーバーに送信しない。サーバーのアクセス
       ログ、リバースプロキシ、WAF、CDN、いずれも鍵を記録できない
  2. **サーバーに鍵を預ける**(TLS 前提で "信頼された中間者" 扱い)
     - 実装は最も単純だが、運営者が復号可能になり、ゼロ知識性が崩壊
  3. **別チャネルで鍵を渡す**(URL と鍵を別メッセージで送る)
     - 攻撃面は最小だが UX が壊れる(通常のユーザーは 1 つのリンクで完結
       することを期待する)
- **決定**: **URL フラグメント方式を採用**(Bitwarden Send 方式)
  - 共有 API は `ciphertext` と `expires_in` のみを受け取り、サーバー側で
    opaque な `id` を発行して返す
  - クライアント(発信者)は 32 バイトの鍵 `k` を `crypto.getRandomValues`
    で生成し、ローカルで AES-GCM-256 暗号化した上で `ciphertext` を POST
    する
  - クライアントは `https://<host>/share/<id>#k=<base64url(k)>` を生成して
    受信者に渡す
  - 受信者ブラウザは `#k=` を location.hash から取り出し、
    `GET /api/v1/share/{id}` で暗号文を取得してローカルで復号する。鍵は
    一度たりともサーバーの可視範囲に入らない
- **根拠**:
  - Lodester のゼロ知識設計(DECISION-006)と整合。運営者が開示請求を受け
    ても共有内容を提示できない
  - Bitwarden Send は 2020 年から実運用されていて、フラグメント方式の
    運用実績が十分にある
  - 白石(セキュリティレビュアー)の M4 レビューで明示的に要求された方式
- **注意事項 / 運用上のリスク**:
  - ブラウザ履歴・ブックマーク・スクリーンショット・ブラウザ同期機能・
    モバイルの URL サジェスト等に鍵ごと残り得る。**UI 側で「このリンクは
    誰でも住所にアクセスできる、取り扱い注意」という明確な警告を出す必要
    がある**(M5 Step 2 の QR 生成でも同じ警告を出す)
  - フラグメントはリファラーヘッダーには含まれないが、リダイレクト先が
    同じ URL を `window.location` として扱うと露出する可能性があるので、
    `/share/{id}` ページは別オリジン・別 iframe を一切含めない純粋な
    静的ページとして提供する
  - `Referrer-Policy: no-referrer` と `Cache-Control: no-store` を受信者
    ページに必ず付与する

### [DECISION-056] 共有リンクの有効期限と削除ポリシー

- **日付**: 2026-04-09
- **コンテキスト**:
  ケイパビリティ URL は「一時的に住所を渡す」用途のため、無期限に残しては
  プライバシーリスクが積み上がる。同時に、UX 的に「7 日後にリンクが切れる」
  のが想定外だと利用者が困るため、合理的な既定値と選択肢を決める必要がある。
- **決定**:
  - **既定有効期限: 7 日**
  - **選択可能な値**: `1h`, `1d`, `7d`, `30d`, `0`(無期限)の 5 つ
  - **無期限(`expires_in=0`)は原則非推奨** — UI 側で「プライバシー上推奨
    しません」とワーニングを出した上で、明示的な確認をとる
  - **期限切れレスポンス**: `GET /api/v1/share/{id}` は 410 Gone を返す
    (404 ではなく 410 にすることで「以前は存在したが今はない」ことを明確化)
  - **物理削除タイミング**: 期限から **30 日** 経過で行を物理削除する。
    これは開示請求や誤削除救済のための最小限の猶予期間として設定する
  - **削除ジョブの実装時期**: M5 MVP では実装せず、手動 SQL(`DELETE FROM
    share_links WHERE expires_at < NOW() - INTERVAL '30 days'`)で運用する。
    Phase 1b で systemd timer または goroutine ベースの自動削除を追加する
  - **最大暗号文サイズ**: 1 件あたり **64 KB**。住所 1 枚(数百バイト)の
    用途では十分、かつ悪意のある大容量アップロードを防げる
- **根拠**:
  - 7 日は Bitwarden Send、Firefox Send、WeTransfer 等の業界既定値と一致
  - 30 日の物理削除猶予は、APPI が要求する「不要となった個人情報の遅滞
    なき削除」と、誤ってリンクを削除した際の復旧ニーズとのバランス
  - 410 Gone を選ぶことで、将来 ID を再利用しないことも明確化できる
  - Phase 1b まで自動削除を先送りするのは、M5 は「機能追加より世に出す
    こと」を優先する(田村指摘)というスコープ方針に従うため

### [DECISION-061] クライアントロードマップ(Phase 2 = ブラウザ拡張を最優先)

- **日付**: 2026-04-09
- **コンテキスト**:
  M4 レビューで黒木(シニアエンジニア)から「クライアント戦略が曖昧なまま
  MVP を出すと、MVP 後の最初の 3 か月で何を作るかが毎回議論になり、開発が
  停滞する。Phase 2 以降のクライアント方針を MVP 前に明文化しておけ」との
  指摘。併せて「ブラウザ拡張 → PWA → ネイティブの順が、Lodester の
  ゼロ知識制約下で最も現実的」との提案があった。
- **選択肢**:
  1. **ブラウザ拡張先行**(Chrome + Firefox / Manifest V3)
     - 住所フォームに遭遇したときに「Lodester から自動入力」を提案できる
     - OAP プロトコルの最大価値(fill once, use anywhere)を最短で体現
     - 暗号処理は既に `lodester-client.js` と argon2-browser で確立済み、
       そのまま拡張のバックグラウンドスクリプトへ流用できる
  2. **PWA / インストール可能 Web アプリ先行**
     - 追加のストア審査がなく配布が最速
     - ただしフォーム自動入力は OS / ブラウザのオートフィル API 越しでしか
       できず、Lodester の価値である「ゼロ知識で住所を出す」体験が弱くなる
  3. **iOS / Android ネイティブ先行**
     - OS レベルの Autofill Provider(iOS 12+/Android Autofill Framework)
       に統合できれば最高の UX
     - ただし審査・署名・更新サイクルのコストが個人 OSS には重い
     - 暗号ライブラリを Swift / Kotlin 用に書き直す必要があり、検証負担が
       二重化する
  4. **当面クライアントは作らない**(Web UI のみで継続)
     - M5 時点の Web UI でも最低限の機能は揃っているため継続運用は可能
     - ただし OAP の競争力(他サービスとの連携)を訴求できず、採用が進まない
- **決定**:
  - **Phase 2 = ブラウザ拡張**(Chrome + Firefox、Manifest V3)を最優先
    で実装する。PWA / モバイルネイティブは Phase 3 以降に回す。
  - M5 では **実装しない**。代わりに `docs/browser-extension-spec.md` を
    M5 Step 7 で起稿し、「どの API を叩き、鍵をどこに置き、どのフォーム
    検出方針を取るか」を仕様レベルで固めておく。
  - Phase 2 の最小機能セット(MMP, Minimum Marketable Product):
    1. Lodester サーバーにログインして Vault を復号(既存 Web UI と同じ
       フロー、鍵はメモリのみ)
    2. 対応フィールド(name / postal-code / address-line1 等の
       autocomplete 属性)に Lodester の住所候補を提案
    3. ケイパビリティ URL(DECISION-055 予定)で受信した共有住所を
       取り込んで、そのまま自動入力に流す
    4. Manifest V3 の service_worker + offscreen document で argon2id と
       AES-GCM を実行(メインスレッドの JS コンテキストに鍵を出さない)
  - Chrome Web Store と addons.mozilla.org の両方に配布。どちらも無料で
    個人開発者が配布可能であることを Phase 2 キックオフ時点で再確認する。
  - ネイティブアプリ(iOS / Android)は Phase 3。OS の Autofill Provider
    統合は Phase 3 のキラー機能として位置づけるが、Phase 2 の使い勝手
    データが出揃うまでは着手しない。
- **根拠**:
  - 黒木の指摘どおり、ブラウザ拡張は「住所を必要とする瞬間(EC チェックア
    ウト)」にその場で提案できる唯一のチャネル。Lodester の "Oh yeah, I
    already have that" 体験はここで最も強く出る
  - 既存の WebCrypto + argon2-browser のコードがほぼそのまま再利用でき、
    開発コストが最小
  - MV3 の service_worker モデルは「鍵を短命に保つ」というゼロ知識設計と

  - ネイティブ先行だと Swift / Kotlin で暗号実装を二重化する負担が
