# 調査 1: Google libaddressinput

## 目的

国際的な住所フォーマットを扱うためのライブラリとして、Google libaddressinput が GDA プロジェクトで採用可能か調査する。

## 主な発見

### libaddressinput の概要

- **配布元**: Google が公開する Apache 2.0 ライセンスの OSS
- **対応国**: 240 カ国以上
- **言語実装**:
  - Java(オリジナル)
  - C++11(Chromium で使用)
- **データ形式**: JSON(直接フェッチまたはバンドル可能)

### 提供されるメタデータ

各国ごとに以下の情報を提供:

- **フォーマット文字列**: ネイティブとローマ字(Latin)の両方
- **必須フィールド**: 国ごとに異なる(例: 日本は郵便番号必須、ドイツは郵便番号必須、アイルランドは郵便番号任意)
- **管理区分名のローカライズ**: 都道府県・州の現地語表記
- **対応言語**: 国ごとの言語リスト
- **特殊フィールド処理**: フランスの CEDEX システム等

### フィールド符号化(8 レター)

| コード | 意味 |
|---|---|
| N | 受取人名(Name) |
| O | 組織(Organization) |
| A | 住所行(Address Lines、複数行可) |
| C | 都市(City / Locality) |
| D | 行政区(Dependent Locality) |
| S | 州・県(State / Province) |
| Z | 郵便番号(Zip / Postal Code) |
| X | 並べ替え用(Sorting Code、CEDEX 等) |

### sub_lnames によるローマ字対応

- 都道府県・州レベルではローマ字表記が自動提供される
- **重要な制約**: **番地レベルのローマ字化は提供されない**(ユーザー入力必須)

## GDA プロジェクトへの含意

### ポジティブ

1. **多国対応が事実上解決**: 240+ カ国を網羅
2. **ネイティブ + ローマ字の二形式が前提**: GDA の設計意図に完全合致
3. **Apache 2.0**: AGPL v3 と互換性あり、商用利用可能
4. **JSON 形式**: プラットフォーム非依存(Web、モバイル、デスクトップ、拡張機能のすべてで利用可能)

### 制約

1. **番地ローマ字化はユーザー入力必須**: 自動翻訳機能はない
2. **行政区名以外のローカライズは限定的**: 街路名等は別途必要

## 結論

**採用決定**。GDA のデータモデルは libaddressinput の 8 レターを採用し、240 カ国対応を最初から組み込む。

ただし、メール型連合モデル採用後の実態としては Phase 1 では日本+主要数カ国に運用上の重点を置く可能性があり、libaddressinput は「全機能対応」から「全機能を持ちつつ実運用は段階的展開」へと位置付けが変わっている。
