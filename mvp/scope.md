# MVP 実装範囲(MVP Scope Definition)

**対象**: Lodester リファレンス実装の MVP(Phase 1a)

**ステータス**: 凍結済み。実装フェーズで段階的に消化する。

---

## MVP の定義

**MVP (Phase 1a)** = 「OAP プロトコルの参照実装として、住所ウォレットのコア機能を含む最初の公開版」

設計は全機能を想定して固めてあるが、実装は段階的(MVP → Phase 1b → Phase 2)。**段階的実装** が基本方針。

---

## 前提条件

### 開発体制

- 開発者: Taketo 1 人
- 時間: 本業傍ら、週 5〜10 時間
- 言語: Go(新規習得)
- **実装支援**: 生成 AI の積極活用(設計が固まっている前提)

### 生成 AI 活用の方針

設計がしっかり固まっている前提で、以下の作業を AI で効率化する:

| 作業 | AI 活用度 |
|---|---|
| ボイラープレートコード | 高 |
| CRUD 実装 | 高 |
| テストコード生成 | 高 |
| API ハンドラ実装 | 高 |
| マイグレーションファイル | 高 |
| ドキュメント・README | 高 |
| i18n 翻訳文 | 高 |
| コードレビューと理解 | 人間必須 |
| セキュリティ監査 | 人間必須 |
| デバッグ | 人間主導 |

### 暗号処理の AI コード取扱注意

Lodester の核心はゼロ知識暗号。AI 生成コードは慎重に扱う:

- AI は暗号ライブラリの誤用を生成することがある(固定 nonce、弱い IV 等)
- 盲信せず、公式ドキュメントと照合
- `gosec` などのセキュリティ静的解析ツールを必ず適用
- 実装サイクル: AI 生成 → 人間読解 → ドキュメント照合 → テスト

## MVP (Phase 1a) 実装機能

### コア機能

| # | 機能 | 実装要素 |
|---|---|---|
| 1 | アカウント作成 | メール検証、マスターパスワード設定、KDF 鍵派生 |
| 2 | ログイン(Bitwarden 方式) | KDF 派生ログインハッシュ、セッション発行 |
| 3 | 暗号化ボールト | AES-GCM-256、暗号文ヘッダ(クリプトアジリティ) |
| 4 | マスターパスワード → 鍵派生 | Argon2id(RFC 9106 準拠) |
| 5 | Person データモデル | 多スクリプト名前、ふりがな、ローマ字 |
| 6 | Address データモデル | libaddressinput 8 レター、多スクリプト対応 |
| 7 | libaddressinput 統合 | 日本 + 主要 5 カ国 |
| 8 | ボールト API | GET/PUT `/vaults/{user_id}` |
| 9 | インスタンス発見 | `/.well-known/oap` + `/health` + `/status` |
| 10 | ランダム形式コード | `A7K2-9MQ3-XC4F@postkey.jp` 形式、Luhn mod 32 |
| 11 | Web フロントエンド(日本語) | 基本 CRUD UI、日本語のみ |
| 12 | 共有機能 | QR コード生成 + 共有 URL |
| 13 | vCard エクスポート | vCard 4.0、`X-GDA-CODE` カスタムフィールド |
| 14 | 公式 Postkey 稼働 | Linode 東京、Caddy、systemd、Reproducible Builds |

| # | 要件 |
|---|---|
| 15 | プライバシーポリシー(弁護士レビュー後公開) |
| 16 | 利用規約(弁護士レビュー後公開) |
| 17 | 安全管理措置の内部文書整備 |
| 18 | 令状対応方針の公開 |
| 19 | Docker Compose セルフホストパック |
| 20 | CI/CD パイプライン(GitHub Actions) |
| 21 | Reproducible Builds |
| 22 | SBOM 生成(syft) |
| 23 | バイナリ署名(cosign) |
| 24 | 監視(BetterStack Uptime、Linode 標準) |
| 25 | バックアップ(Linode Object Storage、日次暗号化) |

### MVP に含まれないもの(意図的な除外)

以下は MVP には含めない。「コア機能でまずローンチし、段階的に成長させる」戦略のため:

- カスタムハンドル → Phase 1b 第 1 波
- TOTP MFA → Phase 1b 第 1 波
- バックアップコード → MFA と同時実装
- 英語 UI → Phase 1b 第 1 波
- WebAuthn / Passkey → Phase 1b 第 2 波

### データモデルは最初から完全版で定義

**重要な補足**(Marcus の推奨): データベーススキーマは MVP から全機能分を最初から定義しておく。機能の追加時に毎回マイグレーションを書くのは負荷が高いため、**データモデル最初から、機能は段階的に**が原則。

具体的には、Person / Address / Vault / Handle / MFA / Capability などのテーブルは最初から存在させ、使わないフィールドは NULL 許可にしておく。

## セーフティネット: チェックポイント方式

### 各マイルストーンでの判定基準

1. スコープ進捗(% ベース)
2. Go 習熟度
3. 本業負荷
4. 精神状態
5. AI 活用度

### 遅延判定基準

スコープ進捗が予定の **70 % 未満** → 次のチェックポイントまでに対応を検討

### 遅延時の対応

| 遅延時期 | 対応 |
|---|---|
| M1 遅延 | Go 学習に時間を追加、実装開始を遅延 |
| M2 遅延 | 暗号化実装の簡素化 |
| M3 遅延 | 共有機能を Phase 1b へ送る(Lite MVP 発動) |
| M4 遅延 | フロント UI を最小化 |
| M5 遅延 | ローンチ延期 |

## Lite MVP 案(さらに最小化が必要な場合)

### Lite MVP 範囲

- コア機能(認証、ボールト、基本 UI)のみ
- 共有機能なし
- vCard エクスポートなし
- 目標: 2〜3 ヶ月でのローンチ

### 発動条件

- M3 時点で進捗 50 % 未満
- 本業の状況が半年以上厳しい見込み
- 精神的に「完成させるのが辛い」状態
